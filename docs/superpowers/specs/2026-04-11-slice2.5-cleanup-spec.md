## Task 1: Fix Merchant Canonicalization Bug

**Priority:** P0 — this is a real bug
**File:** `minx_mcp/core/goal_parse.py`, function `_validate_canonical_names`, lines 291-309

### The Bug

When `resolve=True`, the code calls `_resolve_exact_subject("merchant", name, finance_api)` and then checks `if resolved == name: continue`. This means resolution only passes when the input is *already* the canonical name. If the user provides a merchant alias that normalizes to a different canonical string (e.g., `"SQ *JOES CAFE 1234"` → `"Joe's Cafe"`), `resolved != name`, and validation fails with `"must contain canonical names only"`.

### The Fix

Change the resolve path to accept the resolution result instead of requiring identity:

```python
# Current (broken):
if resolve and finance_api is not None:
    resolved = _resolve_exact_subject("merchant", name, finance_api)
    if resolved == name:
        continue

# Fixed:
if resolve and finance_api is not None:
    resolved = _resolve_exact_subject("merchant", name, finance_api)
    if resolved is not None:
        continue
```

The guard is `resolved is not None` — if `_resolve_exact_subject` found any canonical match (exact or normalized), the name is valid.

### Tests to Add

Add to `tests/test_goal_parse.py`:

1. **Structured create with non-canonical merchant name that resolves** — provide a merchant alias that `_resolve_exact_subject` can normalize. Assert `result_type == "create"`, not `InvalidInputError`.
2. **Structured create with merchant name that does NOT resolve** — provide a completely unknown merchant. Assert `InvalidInputError` with "canonical names only".
3. **Structured create with canonical merchant name** — provide the exact canonical name. Assert `result_type == "create"` (regression test for the happy path).

### Acceptance Criteria

- The resolve path accepts any `_resolve_exact_subject` result that is not `None`
- Existing tests still pass
- New tests cover resolve-success and resolve-failure paths
- `mypy` clean

Move these three functions (and any helpers they call) into `goal_parse.py`. Remove the import from `goal_capture.py`.

**Step 2: Decide what to do with `goal_capture.py`.**

After extracting the functions `goal_parse.py` needs, check what remains in `goal_capture.py`. If nothing else imports from it (grep for `from minx_mcp.core.goal_capture`), delete the file entirely. If other modules import from it, leave it but note the dependency.

The five `_`-prefixed imports from `goals.py` (`_normalize_filter_names`, `_normalize_goal_type`, `_normalize_title`, `_validate_goal_state`) should either:
- Be renamed to drop the `_` prefix in `goals.py` (making them part of the public API), OR
- Be moved into a shared validation module

Prefer renaming — these are validation functions that are clearly part of the goal contract.

### What NOT to Do

- Do NOT rewrite the NL parsing logic. Just move it.
- Do NOT change the behavior of any function being moved. This is a structural refactor, not a feature change.
- Do NOT change the `parse_goal_input` signature or behavior.
- Do NOT refactor `goal_capture.py`'s internal helper structure. Move it as-is.

9. `"retarget my Dining Out goal to $75"` with matching goal → `result_type="update"`, `payload` has `target_value=7500`

14. `_resolve_exact_subject("merchant", "SQ *JOES CAFE", finance_api)` where "Joe's Cafe" exists → returns `"Joe's Cafe"` (normalization path)
15. `_resolve_exact_subject("category", "nonexistent", finance_api)` → returns `None`

## Task 4: Improve `_compute_trend`

**Priority:** P2 — functional but naive, can ride alongside Slice 3
**File:** `minx_mcp/core/trajectory.py`, function `_compute_trend`, lines 169-176

### The Problem

Current implementation compares only first vs last score:

```python
def _compute_trend(scores: list[float]) -> str | None:
    if len(scores) < 2:
        return None
    if scores[-1] < scores[0]:
        return "improving"
    if scores[-1] > scores[0]:
        return "worsening"
    return "stable"
```

A trajectory like `[80, 120, 80]` (low → spike → recovery) returns `"stable"` because first == last. The user confirmed this is a real limitation.

### The Fix

Use a simple linear regression slope (or weighted comparison) instead of first-vs-last:

```python
def _compute_trend(scores: list[float]) -> str | None:
    if len(scores) < 2:
        return None
    n = len(scores)
    # Simple least-squares slope: positive slope = worsening, negative = improving
    x_mean = (n - 1) / 2.0
    y_mean = sum(scores) / n
    numerator = sum((i - x_mean) * (s - y_mean) for i, s in enumerate(scores))
    denominator = sum((i - x_mean) ** 2 for i in range(n))
    if denominator == 0:
        return "stable"
    slope = numerator / denominator
    # Threshold: require at least 5% of mean to declare a trend
    threshold = max(abs(y_mean) * 0.05, 0.01)
    if slope < -threshold:
        return "improving"
    if slope > threshold:
        return "worsening"
    return "stable"
```

This handles noisy data correctly: `[80, 120, 80]` has a near-zero slope → `"stable"` (same as before). But `[80, 120, 130]` → positive slope → `"worsening"` and `[130, 120, 80]` → negative slope → `"improving"`.

### Tests to Update

The existing trajectory tests in `tests/test_trajectory.py` assert trend values. After changing the algorithm, verify:
- Monotonically decreasing scores → `"improving"`
- Monotonically increasing scores → `"worsening"`
- Flat scores → `"stable"`
- V-shaped `[80, 120, 80]` → `"stable"` (not "improving" or "worsening")
- Ascending with noise `[80, 75, 90, 100]` → `"worsening"`
- Single score → `None`
- Two scores, lower second → `"improving"`

### Acceptance Criteria

- `_compute_trend` uses directional awareness (slope or equivalent), not just first-vs-last
- Existing trajectory tests pass (update expected values if the new algorithm changes them)
- New test cases cover noisy/V-shaped data
- `mypy` clean
