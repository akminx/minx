# Slice 6i: FTS5 Memory Search

**Status:** Implemented 2026-04-27
**Depends on:** Slice 6g content fingerprints, Slice 6h secret gate
**Next migration:** `021_memory_fts5.sql`; cleanup migration `025_memory_fts_aliases.sql` refreshes triggers to index `entity_fact.aliases`.

## Goal

Add deterministic, offline full-text search over durable memories so Minx has a reliable retrieval fallback before graph edges, enrichment jobs, embeddings, or agentic investigations depend on memory recall.

## Scope

- Create a SQLite FTS5 table keyed by `memory_id`.
- Keep the FTS table synchronized with `memories` using insert/update/delete triggers.
- Add a `MemoryService.search_memories(...)` API.
- Add a `memory_search(query, scope?, memory_type?, status?, limit?)` Core MCP tool.
- Return stable memory DTOs plus rank/snippet metadata without exposing raw secrets.
- Add a one-shot rebuild script for existing databases.

## Non-Goals

- No embeddings or semantic search.
- No external provider calls.
- No graph traversal.
- No query expansion, LLM rewriting, or harness-side interpretation.
- No search over vault body text; this slice is memory-table only.

## Schema

Migration `021_memory_fts5.sql` creates:

```sql
CREATE VIRTUAL TABLE memory_fts USING fts5(
    memory_type,
    scope,
    subject,
    payload_text,
    source,
    reason,
    tokenize = 'unicode61'
);
```

The FTS `rowid` is always the corresponding `memories.id`. The table stores a denormalized search document because `payload_text` is computed from JSON and cannot be modeled cleanly as an external-content FTS table column.

Triggers:

- `memories_ai_fts`: after insert, write the searchable document.
- `memories_au_fts`: after update of indexed fields, replace the FTS row.
- `memories_ad_fts`: after delete, delete the FTS row.

Indexed fields are `memory_type`, `scope`, `subject`, `source`, `reason`, and a flattened `payload_text`. `payload_text` is generated by SQL triggers with `COALESCE(json_extract(...))` for known payload fields (`value`, `note`, `signal`, `limit_value`, and `aliases`) and by the rebuild script with matching flattening rules.

## Search Semantics

`MemoryService.search_memories` signature:

```python
def search_memories(
    self,
    *,
    query: str,
    scope: str | None = None,
    memory_type: str | None = None,
    status: str = "active",
    limit: int = 25,
) -> list[MemorySearchResult]:
```

Rules:

- `query` is required and whitespace-trimmed.
- `limit` is an integer from 1 to 100.
- `status` defaults to `active`; callers may pass `candidate` for review workflows, or `None` to search all statuses.
- Results join back to `memories`, so the canonical returned memory is still `MemoryRecord`.
- Results order by `bm25(memory_fts)` ascending, then `memories.updated_at DESC`, then `memories.id DESC`.
- Non-`expired` searches apply the same lifecycle guard as memory listing: rows whose `expires_at` is in the past are excluded even if their stored `status` is still `active`.
- Malformed legacy `payload_json` rows are skipped during result materialization so one bad historical row cannot crash a search result page.
- Invalid FTS query syntax raises `InvalidInputError("query is not valid FTS5 syntax")`.
- Empty result is a successful response with `results: []`.

MCP response shape:

```json
{
  "results": [
    {
      "memory": {"id": 1, "memory_type": "preference"},
      "rank": -1.23,
      "snippet": "..."
    }
  ]
}
```

## Security

No new secret-scanning path is needed because 6h blocks or redacts memory writes before persistence. Search results only expose persisted memory DTOs and FTS snippets derived from the same persisted text. The result serializer must not include raw SQLite errors that echo query text.

## Tests

- Migration applies on a fresh DB and creates `memory_fts` plus triggers.
- Creating an active memory makes it searchable by subject and payload value.
- Updating payload removes old terms and adds new terms.
- Rejecting/expiring a memory removes it from default `status="active"` results.
- `status=None` can include non-active rows.
- Scope and memory type filters compose with the FTS query.
- Invalid query syntax returns `InvalidInputError`.
- MCP `memory_search` wraps results in the normal success envelope.
- Rebuild script repopulates missing/outdated FTS rows and is idempotent.
