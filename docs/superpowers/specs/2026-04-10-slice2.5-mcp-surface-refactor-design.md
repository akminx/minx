# Slice 2.5: MCP Surface Refactor

**Date:** 2026-04-10
**Status:** Drafted for review
**Parent:** [Minx Life OS Architecture Design](2026-04-06-minx-life-os-architecture-design.md)
**Depends on:** Slices 1, 2, 2.1

## Goal

Restructure the Core MCP tool surface so it serves as a structured data provider for smart harnesses (Hermes today, others tomorrow), while keeping data-integrity LLM work internal. Remove narrative generation, markdown rendering, and vault writing from Core. Add temporal history and goal trajectory tools that enable harness-level intelligence.

This slice does not change the Finance MCP, domain logic, detectors, or shared platform. It reshapes the boundary between Core and its consumers.

## Why Now

Slices 1-2.1 built the data layer and detection pipeline. But the Core MCP tool surface was designed as if Core would be the one talking to the user — it generates narratives, renders markdown, writes vault notes, and manages clarification dialogues. That is harness work.

Hermes already has skills that do narrative generation, cross-domain reporting, and conversational coaching. The MCP should provide the structured signals those skills need, not duplicate what the harness does better.

The temporal history gap is the most urgent problem. Detectors fire on today's data and the insights get persisted, but nothing exposes historical signals. The harness cannot say "this is the third week dining spiked" because the MCP cannot answer "what fired last week?"

## Success Criteria

- `get_daily_snapshot` returns structured data with no narrative, no vault write, and silently persists detector signals
- `get_insight_history` returns historical detector signals with recurrence counts
- `get_goal_trajectory` returns goal progress across recent periods with trend
- `goal_parse` accepts both natural language and structured input, returns interpretation without managing clarification turns
- `persist_note` lets the harness write vault content through the MCP's atomic writer
- Finance MCP LLM-backed tools accept both structured and natural language input
- All existing tests updated or replaced; no test count regression
- Architecture doc and roadmap reflect the MCP/harness split

## Non-Goals

- This slice does not add new detectors or change detector logic
- This slice does not change the Finance MCP tool surface (except adding dual-path inputs to `finance_query`)
- This slice does not implement durable memory (Slice 6)
- This slice does not update Hermes skills (see Hermes Integration Reference appendix for the migration map)
- This slice does not add new schema migrations

## MCP / Harness Responsibility Split

This is the governing principle for this slice and all future work.

### MCP Owns

- Domain data (CRUD, persistence, migrations)
- Deterministic detectors (threshold-based signal generation)
- Historical insight log (what was detected, when)
- Temporal aggregations (trends, trajectories, week-over-week)
- Read model snapshots (structured data for harness consumption)
- Data-integrity LLM work (classification, parsing, entity resolution)
- Vault persistence mechanics (atomic writes, path safety)
- Goal records, progress computation, and trajectory

### Harness Owns

- When to check in (cron, proactive triggers, scheduling)
- What to tell the user (prioritization, suppression, personalization)
- How to say it (narrative, voice, channel-appropriate format)
- Conversation memory (what was said before, what user responded)
- Cross-domain reasoning (connecting signals the MCP surfaces)
- Coaching and planning (goal guidance, next actions)
- Multi-turn dialogue (clarification, follow-up, confirmation)
- Where to post (Discord channels, vault notes, dashboards)

### The Rule

If the logic depends on the user's data or goals, it belongs in the MCP. If the logic depends on how you are talking to the user, it belongs in the harness.

## Tool Surface Changes

### Replace: `daily_review` → `get_daily_snapshot`

The current `daily_review` tool builds read models, runs detectors, optionally calls an LLM for narrative enrichment, generates a fallback narrative, renders markdown, writes a vault note, and persists detector insights. It returns a protected review projection.

The new `get_daily_snapshot` tool builds read models, runs detectors, silently persists detector insights, and returns structured data. No narrative. No vault write. No LLM call for review enrichment.

```python
@mcp.tool(name="get_daily_snapshot")
async def get_daily_snapshot(
    review_date: str | None = None,
    force: bool = False,
) -> dict:
```

Parameters:
- `review_date`: ISO date string, defaults to today
- `force`: if true, replaces existing detector insights for the date

Response shape:

```python
{

    "timeline": {
        "date": "2026-04-10",
        "entries": [
            {

    "spending": {
```

The `attention_items` field is a convenience list derived from three sources: detector signals (summaries), open loops (descriptions), and off-track or watch-status goals (title + status). Items are sorted by severity (alerts first, then warnings, then info). It gives the harness a quick list of what to surface without parsing the full structured data.

Detector insights are persisted to the `insights` table on every call (using the existing savepoint-based persistence logic). The `force` parameter replaces existing insights for the date, matching current behavior.

If detector persistence fails, the tool still returns the snapshot with a `"persistence_warning"` field rather than raising `ReviewDurabilityError`. The snapshot is always useful even if the DB write fails.

### New: `get_insight_history`

Query historical detector signals to enable temporal reasoning in the harness.

```python
@mcp.tool(name="get_insight_history")
def get_insight_history(
    days: int = 28,
    insight_type: str | None = None,
    goal_id: int | None = None,
    end_date: str | None = None,
) -> dict:
```

Parameters:
- `days`: lookback window (default 28, max 90)
- `insight_type`: filter to a specific detector type (e.g. "finance.spending_spike")
- `goal_id`: filter to insights related to a specific goal
- `end_date`: end of the window, defaults to today

Response shape:

```python
{
    "window": {"start_date": "2026-03-13", "end_date": "2026-04-10", "days": 28},
    "insights": [
        {
            "review_date": "2026-04-10",
            "insight_type": "finance.spending_spike",
            "dedupe_key": "2026-04-10:spending_spike:dining-out",
            "summary": "Spending is up 40.2% versus last week, led by Dining Out.",
            "supporting_signals": ["..."],
            "confidence": 0.85,
            "severity": "warning",
            "actionability": "suggestion"
        }
    ],
    "recurrences": [
        {
            "insight_type": "finance.spending_spike",
            "pattern_key": "spending_spike:dining-out",
            "occurrences": 3,
            "total_periods": 4,
            "description": "3 of last 4 weeks"
        }
    ]
}
```

The `recurrences` field groups insights by pattern (stripping the date prefix from the dedupe key) and counts how often each pattern appeared in the window. This is the primary input for the harness to say "this keeps happening."

Implementation reads directly from the existing `insights` table. No new schema needed.

For `goal_id` filtering: insights related to a goal are identified by dedupe keys containing `goal-{id}` (used by `detect_goal_drift` and `detect_category_drift`) or `goal-risk:{id}` (used by `detect_goal_finance_risks`). The filter matches both patterns.

Recurrence pattern keys are computed by stripping the date prefix (everything up to and including the first colon) from the dedupe key. For example, `2026-04-10:spending_spike:dining-out` becomes `spending_spike:dining-out`. Insights with the same pattern key across different dates are counted as recurrences.

### New: `get_goal_trajectory`

Return goal progress across recent periods to enable trend reasoning.

```python
@mcp.tool(name="get_goal_trajectory")
def get_goal_trajectory(
    goal_id: int,
    periods: int = 4,
) -> dict:
```

Parameters:
- `goal_id`: the goal to query
- `periods`: how many past periods to include (default 4, max 12)

Response shape:

```python
{
    "goal": {
        "id": 1,
        "title": "Reduce dining to $200/wk",
        "goal_type": "spending_limit",
        "metric_type": "sum_below",
        "target_value": 20000,
        "period": "weekly",
        "status": "active",
        "starts_on": "2026-03-15",
        "ends_on": null
    },
    "trajectory": [
        {
            "period_start": "2026-03-17",
            "period_end": "2026-03-23",
            "actual_value": 22000,
            "target_value": 20000,
            "status": "off_track"
        },
        {
            "period_start": "2026-03-24",
            "period_end": "2026-03-30",
            "actual_value": 19500,
            "target_value": 20000,
            "status": "on_track"
        },
        {
            "period_start": "2026-03-31",
            "period_end": "2026-04-06",
            "actual_value": 21000,
            "target_value": 20000,
            "status": "off_track"
        },
        {
            "period_start": "2026-04-07",
            "period_end": "2026-04-13",
            "actual_value": 18000,
            "target_value": 20000,
            "status": "off_track"
        }
    ],
    "trend": "worsening",
    "periods_off_track": 3,
    "periods_on_track": 1
}
```

Implementation: for each past period, compute the goal's actual value using the existing `FinanceReadAPI.get_filtered_spending_total` or `get_filtered_transaction_count` (same logic as `build_progress_for_goal` but applied to historical windows). The trend is derived from the trajectory: improving if the most recent periods show decreasing distance from target, worsening if increasing, stable otherwise.

Period boundaries use the same calendar-aligned computation as the existing `goal_progress.py` — Monday-Sunday for weekly goals, 1st-last for monthly goals. This stays consistent with how `build_progress_for_goal` already works. Periods are computed working backwards from the current calendar period.

### New: `persist_note`

Generic vault write for harness-generated content.

```python
@mcp.tool(name="persist_note")
def persist_note(
    relative_path: str,
    content: str,
    overwrite: bool = False,
) -> dict:
```

Parameters:
- `relative_path`: path within the vault (e.g. "Minx/Reviews/2026-04-10-daily-review.md")
- `content`: markdown content (generated by the harness)
- `overwrite`: if false, fails when the file already exists

Response: `{"path": "/full/path/to/note.md", "created": true}` or `{"path": "...", "overwritten": true}`

Uses the existing `VaultWriter` with its path traversal protection and atomic writes. The `allowed_roots` constraint is widened to accept any path within the vault root (not just `Minx/`). The harness is responsible for choosing appropriate paths — the MCP enforces that writes stay within the vault directory but does not restrict subdirectory structure.

### Refactor: `goal_capture` → `goal_parse`

The current `goal_capture` tool (742 LOC in `goal_capture.py`) does two things:
1. Parses natural language into structured goal intent
2. Manages clarification dialogue (ambiguity resolution, option presentation)

The clarification dialogue is a conversation. Conversations belong in the harness. The MCP should parse and report what it found, then the harness decides how to resolve ambiguities.

The new `goal_parse` tool:

```python
@mcp.tool(name="goal_parse")
async def goal_parse(
    message: str | None = None,
    structured_input: dict | None = None,
    review_date: str | None = None,
) -> dict:
```

Dual-path:
- `message`: natural language, MCP uses internal LLM to parse
- `structured_input`: pre-parsed by the harness, MCP validates

Exactly one of `message` or `structured_input` must be provided.

Response shape (matches current `GoalCaptureResult` — the harness now owns the conversation flow):

```python
{
    "result_type": "create",     # create | update | clarify | no_match
    "action": "goal_create",     # goal_create | goal_update | null
    "payload": {...},            # structured params for goal_create or goal_update
    "assistant_message": "...",  # suggested response text (harness may use or ignore)

    # Only present when result_type == "clarify"
    "clarification_type": "ambiguous_subject",
    "question": "Did you mean Dining Out or Food & Groceries?",
    "options": [...],
    "resume_payload": {...}
}
```

The `structured_input` dict for the structured path must contain at minimum:
- `action`: `"goal_create"` or `"goal_update"`
- `payload`: the same dict that would be passed to `goal_create` or `goal_update`
- `goal_id`: (required for updates) the target goal

When `structured_input` is provided, the tool validates the payload and returns it as a `create` or `update` result without LLM involvement.

The harness receives a `clarify` result, presents the question in its own voice, gets the user's answer, and calls `goal_create` or `goal_update` directly with resolved values. The MCP never manages multi-turn state.

The existing `goal_capture.py` module is simplified and renamed to `goal_parse.py`: the core parsing logic stays, the conversation management (which was thin anyway — the MCP returned structured options rather than truly managing turns) stays as structured output. The main change is renaming, adding the structured input path, and ensuring the module appears in the modified files table.

### Modify: `finance_query` (dual-path inputs)

The existing `finance_query` tool in the Finance MCP server accepts natural language and uses the interpretation pipeline to parse it. Add a structured input path so smart harnesses can skip the LLM.

```python
@mcp.tool(name="finance_query")
async def finance_query(
    # Structured path — harness already parsed
    intent: str | None = None,      # "list_transactions" | "sum_spending" | "count_transactions"
    filters: dict | None = None,    # {start_date, end_date, category_name, merchant, ...}
    # Natural language path — MCP parses via internal LLM
    natural_query: str | None = None,
    # Shared params (both paths)
    review_date: str | None = None,
    session_ref: str | None = None,
    limit: int = 50,
) -> dict:
```

When `intent` and `filters` are provided, skip the interpretation pipeline entirely. Validate the intent and filters, execute the query, return results. `review_date`, `session_ref`, and `limit` work the same as the current tool.

When `natural_query` is provided, use the existing interpretation pipeline (unchanged). The `message` parameter is renamed to `natural_query` for clarity.

Exactly one path must be provided (structured: `intent`+`filters`, or natural: `natural_query`). If both are given, return an error.

## Migration Notes

### For Existing Data

No schema changes. The `insights` table already stores everything `get_insight_history` needs. Historical insights accumulated from previous `daily_review` calls will be queryable immediately.

### For Existing Callers

The `daily_review` tool name is removed. Any caller using it must switch to `get_daily_snapshot`. The response shape changes significantly (no narrative fields). This is a breaking change at the tool boundary.

The `goal_capture` tool name is replaced by `goal_parse`. The response shape is similar but the tool now accepts `structured_input` as an alternative to `message`.

### Skills That Get Thinner

After migration, these skills shrink significantly because the MCP now does the heavy computation:

**finance-insights**: Currently pulls two weeks of summaries and compares them manually. After: calls `get_daily_snapshot` (signals pre-computed) and `get_insight_history` (recurrences pre-computed). The skill becomes "call tools, format for Discord."

**weekly-review**: Currently pulls from 3+ sources and structures the report. After: `get_daily_snapshot` for each day or a weekly window, `get_insight_history` for trends, `get_goal_trajectory` for each goal. The skill focuses on narrative and presentation.

**finance-budget**: Currently manages budget state. After: goals and trajectories come from minx-core. The skill becomes a conversational layer over `goal_list` and `get_goal_trajectory`.

### Skills That Stay The Same

**journal-scan**, **health-log**, **souschef**: These don't touch FinanceHub and don't change until their respective domain MCPs exist (Slices 3, 4, 7).

**finance-scrape**: Browser automation is harness-level. Only the final handoff to import changes (FinanceHub → minx-finance).

### Data Migration

FinanceHub and minx-mcp use separate SQLite databases. Transaction data will need to be migrated from FinanceHub's DB to minx-mcp's DB. This is a one-time operation outside the scope of both this spec and the Hermes skill migration. It should be handled as a separate task before switching Hermes skills to minx-mcp.
