## What Is Minx

Minx is a personal Life OS built as MCP servers.

- Domain MCPs own user facts and domain operations (Finance, Meals, Training).
- Minx Core owns deterministic interpretation (read models, detectors, insight history, snapshots, memory).
- Hermes (or any harness) owns narrative, coaching, conversation flow, scheduling, and wiki maintenance.

Governing rule: **data and deterministic logic live in MCP; conversational policy, scheduling, and LLM prose live in harness.**

## Architecture Decisions

Key architectural decisions that govern future work:

1. **Core/Harness split (Slice 2.5)**: Core provides structured data and tools. Harnesses own orchestration, scheduling, notifications, and LLM-generated prose. This applies to both memory (Slice 6) and autonomy (Slice 8).
2. **Memory tiers (Slice 6)**: Three-tier memory: SQLite for structured facts (Tier 1), SQLite for episodic archives (Tier 2), Obsidian vault as a living wiki (Tier 3, LLM Wiki pattern). Core reads the vault and stores facts. Harness writes wiki pages using LLM via Core tools.
3. **Autonomy split (Slice 8)**: Core provides an audit trail (`playbook_runs`), logging/history MCP tools, and a `playbook://registry` resource. Hermes owns scheduling (uses existing cron), playbook execution scripts, confirmation conversations, and wiki maintenance. No scheduling library in Core.
4. **LLM Wiki pattern (Slices 6+8)**: Inspired by Karpathy — Obsidian is the IDE, the LLM is the programmer, the wiki is the codebase. Minx incrementally builds and maintains structured wiki pages in Obsidian rather than re-deriving knowledge on every query.

### Interpreting mypy output

`uv run mypy` currently reports **0 errors in `minx_mcp/` (production source) and ~216 errors in `tests/`** (snapshot 2026-04-17 post-audit Waves 1–3). That total is expected, not a regression:

- Run `uv run mypy minx_mcp` by itself for the headline production signal — this must always return `Success: no issues found`.
- All test-side errors are the same loose-typing pattern: tests call MCP tools directly and index into the `ToolResponse` dict (typed `dict[str, Any]` by contract), so mypy cannot statically know that `result["data"]["memory"]["id"]` is an `int` on a specific call. Production callers either pass the envelope along unchanged or hydrate it into a typed dataclass, so they don't hit this at all.
- Dominant error codes in `tests/` are `[index]` (~~92), `[arg-type]` (~~59), `[unused-ignore]` (~~20), `[dict-item]` (~~12); none correspond to runtime bugs.
- When a PR or review asks "why is mypy noisy?", the answer is "test-only loose-dict indexing, see `docs/superpowers/plans/2026-04-15-code-quality-cleanup.md` §6.4". The test-only count is expected to trend down opportunistically as that phase lands, not in one sweep.
- The canonical template for tightening a single test file is what `tests/test_hermes_http_smoke.py::_call_tool` does: annotate the tool-call helper's return as `dict[str, Any]` (narrowing from `object`) so downstream indexing into `["data"][…]` is well-typed. Applying this pattern to one file dropped 9 errors from the baseline (205 → 196) on 2026-04-17.

The hard health rule is: `**minx_mcp/` source stays at 0 mypy errors, and every PR that touches a test file should not *increase* the `tests/` count**. Monotonic decrease is the goal; full zero is a nice-to-have, not a gate.

The post-hardening bump from 196 → 210 on 2026-04-17 (and the subsequent audit bump from 210 → 216) is a deliberate, documented exception: the hardening wave added 36+ and the post-audit waves added another ~28 regression tests (race interleavings, snapshot failure-policy, recipe savepoints, import-size limits, symlink escapes, log redaction, vault file locks, BaseService asyncio-task isolation, recipe reconcile, LLM error envelopes, DB-lock-during-LLM-await probes, `log_meal` atomicity) using the same loose-dict-indexing pattern the rest of the suite uses. The `ToolResponse[T]` generic / per-tool Pydantic DTO migration that would fix these at the source is explicitly deferred as a standalone refactor (see `docs/superpowers/plans/2026-04-15-code-quality-cleanup.md` §6.4) rather than being gold-plated into a hardening wave. The ratchet's intent — "don't casually grow test-side debt" — is preserved: we prefer this bump over sacrificing race-condition coverage.

## Post-Upgrade Operator Steps

After pulling the 2026-04-17 hardening wave (or any later revision that has not yet been deployed), run the following one-shot maintenance command before the next finance import:

```bash
python -m scripts.rebuild_finance_dedupe  # uses settings default DB path
# or: python -m scripts.rebuild_finance_dedupe /path/to/minx.db
```

This rebuilds `finance_transaction_dedupe` using the new merchant-aware fingerprint algorithm. Without it, re-importing the same source CSV/PDF after the upgrade would produce duplicate transactions because old fingerprints no longer match fresh hashes. The script is idempotent and savepoint-wrapped — running it more than once is safe and a no-op after the first successful run.

## Known Breaking Surface Changes (2026-04-17)

- **Finance report JSON payloads**: weekly/monthly report `summary` objects changed monetary fields from float dollars (e.g. `total_amount: -12.50`) to integer cents (`total_amount_cents: -1250`), and the `import_preview` `amount` field became a two-decimal string (`"-12.50"`) via `format_decimal_cents`. Any consumer outside the in-repo smoke tests that reads those fields by name must be updated in lockstep. Hermes integration was spot-checked via `tests/test_hermes_http_smoke.py`; no Hermes-side consumer reads those fields today, but the schema change is documented here so future consumers know to expect cents integers.
- **Meals MCP default `goal`**: omitting `goal` on the `nutrition_profile_set` tool used to implicitly mean `fat_loss`; it now means `maintenance` (no calorie multiplier applied). This is the safer default per the new `GOAL_CALORIE_MULTIPLIERS` semantics. Callers that want the old behavior must explicitly pass `goal="fat_loss"` (preserved as a legacy alias for `cut`).
- **Memory payload schemas**: `memory_create` and `memory_update_payload` now enforce per-`memory_type` payload shapes via pydantic models with `extra="forbid"` (see `minx_mcp/core/memory_payloads.py`). Known types are `preference`, `pattern`, `entity_fact`, `constraint`; unknown `memory_type` values still pass through unchanged for forward compatibility. Clients that previously stuffed arbitrary keys into `payload` will now receive `INVALID_INPUT` with the offending field named. `ingest_proposals` skips bad proposals with a WARNING log rather than aborting the batch.
- **LLM error contract**: upstream LLM failures (provider HTTP errors, malformed JSON, schema validation failures inside `interpretation/runner.py`) now surface with `error_code="LLM_ERROR"` instead of the generic `INTERNAL_ERROR`. Harnesses and clients that branch on error codes should add a case for `LLM_ERROR` (typically: retry once, then surface to the user as an upstream-model problem rather than a server bug).
- **`meals.recipe_orphaned` event**: new event emitted by `MealsService.reconcile_vault_recipes()` when a row's `vault_path` .md file is missing on disk; `vault_path` and `vault_synced_at` are nulled but the row itself (and any `meals_meal_entries.recipe_id` foreign keys) is preserved. Consumers of the event bus should add a handler or explicitly ignore `meals.recipe_orphaned`.
- **Rejected memory TTL**: `memory_reject` now sets `expires_at = now + 30 days`; `memory_list` / `get_pending_memory_candidates` call `prune_expired_memories()` before returning, so any tooling that relied on expired-rejected rows remaining visible indefinitely must instead query with `status="rejected"` within the 30-day window.
